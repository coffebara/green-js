1. 작업폴더(작업트리) f1.txt f2.txt 추적되지 않는 Untracked files 한번도 버전관리 하지 않은 파일이다. 2. <스테이지> staging area는 commit을 하기 전에 commit할 파일들을 골라놓는 곳입니다. 그리고 staging area에 파일 넣는 행위를 staging이라고 합니다. git add 명령어로 staging 할 수 있습니다. 3. <저장소> repository 는 commit 된 파일의 버전을 모아놓는 곳입니다. repository의 실체를 구경하고 싶으면 작업폴더안에 숨겨져 있는 .git 폴더 열어보면 됩니다. git commit -m '첫파일을 만들었음' git status 상태창 git log --all --online commit된 기록을 한눈에 파앍하고 싶으면 $ git remote add origin https://github.com/allready-dawn/fullStack.git git push -u 원격저장소주소 main git push -u origin main : 좋아 커밋상태에 있는 내용들을 원격저장소로 밀어보내줄게! - 로컬저장소의 main 브랜치를 원격저장소에 올리라는 뜻입니다. 다른 브랜치도 올릴 수 있음 참고로 -u 옵션은 방금 입력한 주소 기억해두라는 뜻입니다. 다음부터는 주소를 길게 입력안하고 git push만 입력해도 잘됩니다 원격 repository 주소는 이렇게 http:// 부터 시작해서 .git으로 끝납니다. 이제부턴 git push 만 해도 기억해서 잘 들어가게 된다. --------------------------------------------------------------------- 앞에서 깃에서 사용하는 명령은 리눅스 기반으로 하고, 윈도우에서는 깃 배시 프로그램을 거쳐서 깃 명령을 사용한다고 설명했지요? 윈도우에서 깃을 사용할 때는 주의해야 할 점이 있습니다 윈도우의 줄바꿈 문자와 리눅스 줄바꿈 문다가 다르다는 것입니다. 개행문자 또는 eol(end of line)이라고 부르기도 하는 줄바꿈 문자란 텍스 문서에서 Enter를 눌렀을 때 그 위치에 삽입되는 문자입니다 윈도우에서 문자를 저장했을 경우에는 줄이 바뀌는 자리에 우리 눈에 보이진 않지만 CR 문자와 LF 문자가 삽입됩니다 (합쳐서 CRLF 문자라고 부릅니다) 리눗그와 맥에서는 문서를 저장할 때 줄이 바뀌는 자리에 LF문자가 삽입된니다 그래서 윈도우에서 게스트 문서를 스테이지에 올릴 때는 'WARING:LF will be replaced by CRLF in hello.txt' 같은 경고 메세지가 나타납니다. 이 메세지는 깃에서 자동으로 텍스트 문서의 CRLF 문자를 LF 문자로 변환해서 커밋할 것이라는 의미다 commit 이라는 항목 옆에 영문과 숫자로 된 긴 문자열이 나타나는데 이것을 커밋 해시(commit hash)또는 깃 해시 (git hash)라고 합니다. 커밋을 구별하는 아이디라고 생각하면 쉽습니다 커밋 해시 옆에 있는(HEAD -> main)는 이 버전이 가장 최신이라는 표시입니다 폴더하나 만들고 git init 해준다 github.com은 이제 기본 브랜치 이름을 master가 아니라 main으로 사용하라고 강요합니다. 그래서 우리 로컬 작업폴더에 있는 기본 브랜치 이름도 main으로 변경해줍시다. git branch -M main <<- 기본 브랜치가 main으로 바뀌게 된다 git add . git commit -m 'a파일을 만들었음' git push -u 원격저장소주소 main - 로컬저장소 main 브랜치를 원격저장소에 올리라는 뜻입니다. 다른 브랜치도 올릴 수 있음 참고로 -u 옵션은 방금 입력한 주소 기억해두라는 뜻입니다. 다음부터는 주소를 길게 입력안하고 git push만 입력해도 잘됩니다
